# v0.2.x Tutorial Track — Scoping Plan
**Date:** 2026-05-01
**Status:** Active
**Supersedes:** —

## Goal

Add a `docs/tutorial/` track for v0.2.x that shows the workflow's individual functions running on real material, so that two reader profiles can self-serve:

1. **Academic-skeptic** — has not used Claude Code, will not run an "agent writes my paper" pipeline, but will watch a `theorist` propose a model variant or a `designer` flag a confound and decide whether they'd accept the output. Needs to see what each function *actually produces* before trusting it.
2. **No-prior-Claude-Code reader** — has never seen an agent dispatch, plan-mode handoff, or hook firing. Needs to see what those look like in the wild, not just read about them.

The tutorial deliberately **de-emphasizes orchestrator / `/new-project` end-to-end mode**. Real research at quality bar will not be produced via end-to-end orchestration. Tutorial leads with standalone-skill demos; orchestration gets one short page noting it exists but pointing readers at standalone skills as the real day-to-day surface.

## Approach

### Pedagogical structure

Function-by-function demos, threaded loosely through one running scenario per overlay so readers see how outputs from one skill feed the next without committing to full orchestration.

**Behavioral scenario (priority — no other public template covers this stack):**
"You are prepping a behavioral pilot on [topic TBD — probably something simple like risk-elicitation calibration or a moral-judgment task]." Demos pull in `/discover` (literature), `/theory` (formal model with testable predictions), `/design experiment` (14-step inference-first checklist), `/preregister`, `/qualtrics` and/or `/otree` (implementation), `/challenge --design` (devil's advocate), `/humanize` (instructions polish). Each demo is self-contained — reader can skip to the one they need.

**Applied-micro scenario:**
"You are evaluating a public-data identification strategy on [topic TBD]." Demos pull in `/discover`, `/strategize` (identification memo), `/balance`, `/event-study`, `/challenge --identification`, `/analyze`. Lower priority than behavioral since strategist-style identification reviewing is more familiar territory in econ.

### What each demo page contains

| Section | Content |
|---|---|
| Setup | One-paragraph scenario, what files / data exist before running the skill |
| Invocation | Exact command the reader types |
| What happens | Plain-prose narration of agent dispatch, plan mode, critic review, with screenshots or transcripts of real output |
| The actual output | The artifact produced (memo / model / design doc / pre-registration / Qualtrics survey / table / figure / review report), inline or linked |
| Critique-the-output prompt | "Here's what an experienced [behavioral economist / applied microeconomist] should check before accepting this." Calibrates trust. |
| What this skill does *not* do | Explicit limits — e.g., `theorist` won't decide whether a model variant is publishable, only whether it is internally consistent and matches stated assumptions. |

### What stays out of scope for v0.2.x

- End-to-end orchestrator demos (`/new-project`). One short page acknowledging it exists, pointing readers at standalone skills. Not a tutorial.
- Video / screencast equivalents. Markdown + transcripts only for v0.2.x; revisit if reader feedback asks for video.
- A complete worked paper. Demos use plausible scenarios but do not produce a publication-bar end product — that misframes what the workflow does.
- Customization deep-dives beyond what `docs/customization/adapting-claude-md.md` already covers.

### Demos to include (priority-ordered)

**Behavioral overlay (priority track):**

1. `/theory develop` — given a one-paragraph research question, produce a formal model + predictions. Show the theorist proposing assumptions, deriving propositions, and `theorist-critic` flagging an assumption gap.
2. `/design experiment` — same scenario, design proposes a treatment-arm structure, runs the 14-step inference-first checklist, `designer-critic` flags a confound or power problem.
3. `/preregister` — generate an AsPredicted or OSF pre-registration from the design doc. Show what fields get auto-filled vs which require user judgment.
4. `/challenge --design` — devil's-advocate review of the design memo. Reader sees what objections the workflow raises *before* a real referee would.
5. `/qualtrics create` and/or `/otree create` — generate implementation from design. Show one of the two; the other gets a shorter sibling demo.
6. `/humanize` — strip AI-prose from subject instructions before deployment. Demonstrates universal humanizer on slide-register text.

**Applied-micro overlay:**

7. `/strategize` — given a research question + dataset description, produce identification memo. Show `strategist-critic` flagging a parallel-trends assumption.
8. `/balance` and `/event-study` — generate balance table and event-study plot from a public dataset.
9. `/challenge --identification` — devil's-advocate review of identification strategy.

**Universal (smaller demos referenced by both tracks):**

10. `/discover lit-review` — bibliography production from a topic.
11. `/analyze` — end-to-end script generation for a small public dataset, with `coder-critic` review surfacing.
12. `/review --paper` — take an existing draft, produce writer-critic + domain-referee + methods-referee scores.
13. `/humanize` (already in behavioral track but also documented universally as the cover-letter / response-letter use case).

### Connecting tissue

Each demo links to the next-natural-skill ("after `/theory`, you probably want `/design`") without forcing readers through the chain. Readers can land on a single demo via search/SEO and get full value from it standalone.

## Verification steps

For each demo page, before merge:

1. Author runs the skill with the documented invocation, on a fresh scenario file.
2. Real output captured into the demo (transcript or screenshot — not synthesized).
3. Reader pre-test: hand the page to one academic who has not used the workflow. Watch for what confuses them. Revise.
4. Hook compliance: any paper citations in the demo prose are grounded in `master_supporting_docs/literature/reading_notes/` per `primary-source-first.md`, or escape-hatched with `<!-- primary-source-ok: ... -->` if illustrative.
5. Anti-AI-prose pass: each demo runs through `/humanize docs/tutorial/<file>.md` before merge (dogfood the skill).
6. Compile / link check: every cross-reference in `docs/tutorial/` resolves; `docs/README.md` and overlay README pointers all reach valid files on their respective branches.

## Open questions

1. **Scenario topics.** Behavioral track needs one concrete research question that is plausible enough for `theorist` and `designer` to produce non-trivial output, but not so specific that it commits the tutorial to a particular paradigm. Candidates: risk-elicitation calibration, moral-judgment task, simple game-theoretic test. Decide before writing demo 1.
2. **`/qualtrics` vs `/otree` for the implementation demo.** Pick one for v0.2.0 (faster ship); the other goes to v0.2.1. Recommend `/qualtrics` first since survey experiments are the larger user base in behavioral econ.
3. **Public dataset for applied-micro track.** Needs to be small enough that scripts run in seconds in a tutorial setting, real enough that `strategist` produces a non-trivial memo. CPS extracts and Census TIGER files are candidates.
4. **Where to host transcripts / screenshots.** In-repo PNGs bloat git history; external host (CDN / Cloudflare R2) needs an account decision. For v0.2.0, propose: text transcripts inline; screenshots only where strictly needed, kept in `docs/tutorial/_assets/` and pruned aggressively.
5. **Scope of orchestrator-mode acknowledgment page.** One short paragraph, or a longer "why we don't lead with this" section that engages the academic skeptic directly? Tentative: longer version, since the tutorial's positioning depends on the reader believing standalone skills are the real surface.
6. **CHANGELOG note for v0.1.0.** Separate cleanup item: v0.1.0 release shipped the `/humanize` skill + `anti-ai-prose` rule but the v0.1.0 CHANGELOG entry doesn't list them (they landed 2026-04-29, after the CHANGELOG was finalized 2026-04-28). Decide: amend the v0.1.0 entry (post-hoc but accurate), or note in v0.2.0 entry as "shipped in v0.1.0, documented here." Recommend the latter — CHANGELOG entries should be immutable once tagged.

## Sequencing

v0.2.0 minimum-viable tutorial: behavioral track (6 pages) + universal core (3 pages) + tutorial README. ~10 pages. Applied-micro track ships in v0.2.1.

Re-scope after first reader pre-test (step 3 above) on the first behavioral demo. If demo-1 doesn't land, the rest of the structure is wrong.
