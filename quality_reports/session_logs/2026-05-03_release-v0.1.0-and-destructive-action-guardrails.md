## Design Decisions

| Decision | Alternatives Considered | Rationale |
|---|---|---|
| All-caps citation filter as `len(token) >= 3 and token.isupper()` | (a) Add each status marker to NEVER_SURNAMES; (b) Match a fixed enum of common markers | Status markers are an open class — every project invents new ones (SHIPPED, READY, KILLED, DEPRECATED). All-caps filter catches the class with zero precision cost: real surnames in academic prose are never written ALL-CAPS. Side effect catches BLS/OECD/USDA — also intentional (corporate-author data citations aren't framing claims about research papers). |
| Hook name: `destructive-action-guard.py` | Spec said `destructive-on-shared-block.py` | Spec's name suggests path-conditional only, but the hook handles two tiers (always-blocked + path-conditional). Renamed for accuracy. |
| Shared-storage prefix: anchor on canonical macOS mount points | Spec said "any path containing `/Dropbox/`" | Substring match would block `rm -rf /tmp/dropbox-clone/foo` falsely. Anchored on `~/Library/CloudStorage/Dropbox*` and similar. |
| `git rebase` non-resumption blocking | Loosen to only `--root` and `--onto` | Spec-strict for v1 — easier to relax than tighten. Resumption forms (`--continue`, `--abort`, `--skip`, `--quit`, `--show-current-patch`, `--edit-todo`) explicitly allowlisted. |
| `realpath`-aware path resolution + `cd X && Y` parsing | Substring match on raw command | Spec test-case 16: `cd ~/foo && rm -rf bar` where `~/foo` symlinks into Dropbox must block. Realpath through symlinks during cd-chain consumption handles this. macOS-specific gotcha: `/var` → `/private/var` symlink also needs realpath on the prefix list (caught by initial test failures, fixed by realpath-ing both sides of the prefix comparison). |
| Tier-3 settings.json gate: token-based detection | Substring detection on raw command string | First attempt was substring-based; over-fired on its own commit message because the message body documented `> .claude/settings.json` as an example. shlex preserves quoted argument bodies (commit messages, heredocs interpolated via command substitution) as opaque single tokens — so token-walk for the `>`/`>>`/`tee`/`sed -i`/`mv`/`cp`/scripted-write patterns immunizes against quoted-string false positives. |
| Bypass mechanism: `BYPASS_SHARED_GUARD=1` env-var prefix | (a) Permission allowlist; (b) Per-command "are you sure?" prompt; (c) No bypass at all | Env-var prefix is visible in shell history (audit trail), the audit log records `bypass: true`, and the user has to type it deliberately. Permission allowlist would be too coarse; prompts can't be programmed (this is hook output, not interactive). |
| Audit log at `.claude/state/destructive-action-guard.log` | (a) No log; (b) System-level logging | JSONL append-only with 1MB rotation gives both verification (subagent coverage assumption can be checked retroactively) and safety net for any unexpected behavior. Internal logging-failure must never break the hook (try/except wrapper). |
| Trust the doc-researcher answer on subagent hook semantics + add log-based safety net | (a) Empirical test by adding a temp logging hook; (b) Block until verified | The doc-researcher returned a confident, sourced answer from `code.claude.com/docs/en/hooks` claiming hooks fire on subagent Bash with `agent_id` field present. Empirical verification would have required modifying settings.json (gated) and dispatching a subagent — high friction. Compromise: trust the doc + audit log lets us verify retroactively on any subagent dispatch. |
| Plan doc separate from feature commit | Squash everything into one commit | Plan docs are workflow-internal; the feature commit should be the atomic infrastructure change. Easier to cherry-pick the feature without dragging the plan. |
| Token-based detection added 5 regression test cases for false-positive class | Just fix the bug and move on | Token-based architecture is the right invariant; locking it with explicit "git commit -m with write-pattern strings in message MUST allow" tests prevents future regression on the same class. |

## Learnings & Corrections

- **[LEARN:hooks]** Token-based detection on Bash command strings is fundamentally more robust than substring detection. shlex.split preserves quoted argument bodies as single tokens, which means quote-aware detection comes for free if you walk tokens instead of regex-matching the raw command string. Apply this lesson to future Bash hook design.
- **[LEARN:macos]** macOS path resolution uses `/var` → `/private/var` and similar symlinks. Any prefix-match comparison after `realpath()` must realpath BOTH sides. Caught this via test failure on the symlink test case.
- **[LEARN:protect-files]** A hook with matcher `Edit|Write` literally cannot observe Bash file writes. Protect-files.sh-style hooks need a Bash-side companion to be complete protection. The `protect-files.sh` + `destructive-action-guard.py` pair now covers both surfaces for settings.json.
- **[LEARN:cherry-pick]** Use `git cherry` (not `git log --oneline branch..main`) to identify TRULY missing commits between branches. Cherry shows `+` for missing-by-patch and `-` for already-applied-via-different-SHA. After bulk-sync work, most "missing" commits are content-equivalent and don't need re-applying.
- **[LEARN:false-positive-cost]** A false-positive in a security-relevant hook costs more than a false-negative in some ways: the user has to use the bypass for legitimate work, which trains them to bypass habitually. The Tier 3 commit-message false positive was caught immediately by the hook blocking its own commit; if it had been more subtle, it could have eroded the bypass discipline. Token-based detection eliminated the false-positive class structurally.
- **[LEARN:plan-first]** Plan-first surfaces design decisions before code. Three meaningful deviations from the spec (hook name, shared-prefix scoping, rebase blocking) and five risks (subagent coverage being the load-bearing one) all came out of writing the plan, not the implementation. The user's explicit "wait for approval" instruction was load-bearing.
